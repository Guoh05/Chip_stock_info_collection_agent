## Common patterns (apply to every source)

Before the per-source notes, three rules every scraper must obey:

### Rule 1 — Canonical schema mapping

Every per-variant `extracted` dict MUST emit these five fields. Empty values are `0` / `null` / `[]` (never absent keys):

| Field | Semantics |
|---|---|
| `stock_now_qty` | 现货 quantity (immediately shippable) |
| `stock_now_ship_text` | Visible 发货时间 text — **only if the page actually shows it**, else `None` |
| `stock_future_qty` | 期货/在途/factory order quantity. `null` for unbounded factory order. |
| `stock_future_ship_text` | Visible future-ship text, else `None` |
| `stock_breakdown` | `[{label, warehouse, quantity, ship_text, [moq], [ships_from], [note]}, …]` — one row per pool the **page itself exposes** |

Site-native fields preserved alongside under `site_*` keys (`site_global_stock`, `stock_gd_warehouse`, etc.).

### Rule 2 — No fabricated labels

**If a label isn't visible on the rendered product page, don't put it in the JSON.** Period.

This is non-obvious because every distributor's API / SSR blob carries internal field names (`gdWarehouseStockNumber`, `cms-leadtime-ship-estimate`, `factory_stock`) that look like data but are either tooltip strings, CMS keys, or accounting buckets the user can't see. The audit rule: open the product page screenshot, grep the rendered HTML for the literal label string — if it's not there, the scraper must not surface it as a UI-style row.

History of audit corrections (see `scraper_report_v3.md` § D for full list):

- LCSC: removed fabricated `广东仓` / `江苏仓` / `SMT扩展库` warehouse rows.
- Digikey: removed `DigiKey 美国仓` warehouse, `下单后立即发货` ship_text, `工厂期货` row.
- RSONLINE: removed hardcoded `RS 欧时仓` warehouse label.
- Future: dropped `Future Electronics (Singapore)` row (duplicate of global).

### Rule 3 — No silent wrong-variant fallback

When a multi-variant source returns N candidates and **none exactly matches the input MPN**, picking the highest-stock candidate has historically led to silent disasters (LCSC `LTW-M140SXT57-PA` → JST connector, RSONLINE `EMW3080` → ST Discovery kit, Rochester `ESP32-WROOM-32E-N4` → Skyworks Si532).

**Guard pattern (now applied uniformly):**

```python
target_alnum = re.sub(r"[^A-Za-z0-9]", "", input_mpn).upper()
def fuzzy_ok(ret_mpn):
    r = re.sub(r"[^A-Za-z0-9]", "", ret_mpn or "").upper()
    return target_alnum in r or r in target_alnum   # bidirectional substring
```

If neither an exact match nor a fuzzy-substring match exists, return `no_results`. **Never** fall back to "the variant with the most stock."

### Rule 4 — Optional unified fields (cross-source canonical)

In addition to the five mandatory canonical fields (Rule 1), scrapers may populate optional cross-source fields that mirror the API track schema. These show up as columns in the warehouse-exploded `batch_index.csv` and as rows in per-cell `summary.md`.

**`packaging_option`** — shipping / break form (e.g. Tape & Reel / Cut Tape / Tray / Tube / 编带 / 管装 / 散料). **Original site wording preserved verbatim — do NOT translate or canonicalise the value**, leave empty when the source does not publish it. Per-source population (shipped 2026-05-27):

| Source | Populates `packaging_option`? | From | Native values |
|---|---|---|---|
| LCSC | ✅ cell-level | `productRecord.productArrange` | `编带` / `管装` / `散料` (CN) |
| Digikey | ✅ cell-level | `priceQuantity.pricing[0].packaging` | `Tape & Reel (TR)` / `Cut Tape (CT)` / `Digi-Reel®` |
| Future | ✅ cell-level | text-line scrape from `"N per <form>"` / `Available Packaging` row | `Tray` / `Reel` / `Tube` |
| Rochester | ✅ cell-level | spec table `Packaging Type` row | `Tray` / `Reel` / `Tube` etc. |
| bom2buy | ✅ **per-warehouse row** | `.td-stock` cell text tail propagated through `stock_breakdown[i].packaging_option` | `Tray` / `Tape & Reel` / `Cut Tape` / `卷盘` (mixed CN/EN; different distributors use different wording) |
| HQEW | ❌ — cell-level `""` | n/a | column 7 is chip case (e.g. `SOT-223`), not shipping form |
| ONEYAC | ❌ — cell-level `""` | n/a | not published on product page |
| ICKEY | ❌ — cell-level `""` | n/a | not published |
| RSONLINE | ❌ — cell-level `""` | n/a | `packType` = `FINISHED` / `BULK` — RS-internal classification, not shipping form like Tape/Reel; left empty per Rule 2 "no fabricated labels" |

Cell-level value lives in `extracted.packaging_option`. For bom2buy the cell-level value is empty by design; the per-row value lives in `extracted.stock_breakdown[i].packaging_option` and is what the warehouse-exploded export reads. Both `batch_scraper_test.py` (8 main sources) and `_merge_bom2buy_into_batch.py` write the `packaging_option` column in `batch_index.csv` between `currency` and `datasheet_url`, matching the API track schema position.

## 1. LCSC (立创商城, szlcsc.com)

**Script:** `scraper/scripts/scrape_lcsc_v3.py`

### Engine + why

Playwright **Chromium** launched with `args=["--headless=new", "--disable-blink-features=AutomationControlled"]`.

| Engine tried | Result | Why |
|---|---|---|
| `curl_cffi` chrome131 | Returns HTML but the right-panel hydrates client-side | Tier-price panel is JS-only, never present in raw HTML |
| Playwright `headless=True` (legacy) | Right panel stays as `animate-pulse` skeleton | Modern szlcsc detects legacy headless and suppresses hydration |
| Playwright `--headless=new` | ✅ works | `--headless=new` reports as modern Chromium with full rendering |

### URL patterns

- Search: `https://so.szlcsc.com/global.html?k=<MPN>`
- Detail: `https://item.szlcsc.com/<item_id>.html` (e.g. `https://item.szlcsc.com/250032.html`)

### Detail-page reach

1. Goto search URL → wait DOM ready.
2. Collect every result link tagged `s_s__` (search-source = keyword). **Drop** links tagged `from=kw` (those are recommendation / cross-sell carousel, not the actual search results).
3. For each `s_s__` link, open the detail page in the same context.

### Two parallel data sources on each detail page

LCSC ships a **server-side `__NEXT_DATA__` envelope** AND a **client-rendered right panel**. You need both:

| Source | What it carries | Why |
|---|---|---|
| `__NEXT_DATA__.props.pageProps.webData` (SSR) | `productRecord`, `brandVO`, `paramList`, stock numbers (`gdWarehouseStockNumber`, `jsWarehouseStockNumber`, `smtStockNumber`, `totalStockNumber`, `usableTransitNum`, `isDisplayUsableTransitNum`), `pdfFileDetailVO` (datasheet URL), 12+ product parameters | Reliable structured data |
| DOM right-panel (locate via "梯度" and "库存总量" text anchors, anc up 8 levels) | Price tiers (6+ rows: qty range × unit price), user-facing ship-text strings ("最快4小时发货", "3个工作日内发货") | Tiers don't exist in SSR — they hydrate via XHR after page load |

### Page-element semantics

| Field | DOM source | Means | Use as | Discard? |
|---|---|---|---|---|
| `gdWarehouseStockNumber` (API) | SSR | Guangdong warehouse stock (LCSC's PRC HQ) | Goes into `site_*`; do NOT label as `广东仓` row in breakdown (page UI does not show this label) | — |
| `jsWarehouseStockNumber` (API) | SSR | Jiangsu warehouse stock | Same — `site_*` only | — |
| `smtStockNumber` (API) | SSR | SMT subsidy / 扩展库 (internal accounting) | `site_*` only | — |
| Visible "**现货 N**" badge | DOM | Aggregated total user sees | The canonical `stock_now_qty` | — |
| `usableTransitNum` (API) | SSR | In-transit incoming (only when `isDisplayUsableTransitNum=true`) | `stock_future_qty` + 在途 breakdown row | When `isDisplayUsableTransitNum=false`, drop — page doesn't expose it |
| Visible "**起订量：N 个**" span | DOM (`<span class="text-[#54657D]">起订量：5 个</span>`) | Real buyer-facing MOQ. Often differs from API's `minBuyNumber=1`. | `min_order_qty` (canonical) | — |
| `minBuyNumber` / `minWholeNumber` (API) | SSR | Internal min-buy / pack quantities | `site_*` only | — |
| Tier price table | DOM right-panel | Volume pricing | `prices[]` | — |
| `paramList` | SSR | Spec parameters | `parameters[]` (use keys `parameterName` / `parameterValue` — NOT the legacy `paramName` keys) | — |

### Pitfalls

- **Modern headless ≠ legacy.** Must use `--headless=new`. Without it, the right panel never hydrates.
- **Per-warehouse API breakdown is not page truth.** LCSC's API splits stock across 广东仓 + 江苏仓 + SMT扩展库, but the UI shows only the aggregate. Don't emit per-warehouse rows in `stock_breakdown`; the aggregate `现货` row (warehouse=None) is what matches the screenshot.
- **Two MOQ concepts.** `min_buy_number` (API, often 1) ≠ `min_order_qty` (visible "起订量：N 个", often larger). The visible value is what a buyer must use; expose both for transparency.
- **Multi-variant pickup.** A search for `LTW-M140SXT57-PA` can return 30 keyword-tagged results, most unrelated. Apply the Rule 3 fuzzy guard.

### Known transient failure

`[WinError 5] 拒绝访问.: '_tmp_NNN' → …` from a Windows Playwright temp-cleanup race. ~5 % of cells on a 51-chip sweep. Rerun usually clears.

## 2. Digikey (得捷电子, digikey.cn)

**Script:** `scraper/scripts/scrape_digikey.py`

### Engine + why

Playwright **Chromium** with `playwright-stealth`. The `.cn` site fronts a Cloudflare JS challenge (zh-CN title: "请稍候") that resolves in ~10 s under a real browser.

| Engine tried | Result | Why |
|---|---|---|
| `curl_cffi` chrome131 | 403 Cloudflare challenge page | CF needs JS execution |
| Playwright Chromium (no stealth) | Detected, infinite "请稍候" | webdriver fingerprint |
| Playwright + `playwright-stealth` | ✅ works | Stealth hides webdriver markers |

### URL patterns

- Search: `https://www.digikey.cn/zh/products/result?keywords=<MPN>`
- Detail: `https://www.digikey.cn/zh/products/detail/<mfr-slug>/<MPN>/<digikey-id>`

For exact-match queries, the search URL **redirects** to the detail URL. For ambiguous queries, the search page lists multiple `/products/detail/...` links.

### Detail-page reach

1. Goto search URL with `wait_until="domcontentloaded"`.
2. **Cloudflare wait loop.** Poll page title for `'Just a moment' in title` (en) AND `'请稍候' in title` (zh-CN). Pair the title check with `len(html) > 50_000` as a real-page signal. **Don't trust "MPN in HTML"** — Cloudflare embeds the original keyword in the challenge body, false positive.
3. If redirected to a detail page, scrape it. Else click first `/products/detail/` link.
4. **Second Cloudflare wait** after detail-page navigation (it sometimes re-challenges, ~20–30 s). Guard with `cloudflare_persisted_on_product` to avoid silent fall-through.

### Data extraction

All structured data lives in `__NEXT_DATA__.props.pageProps.envelope.data`:

- `productOverview`: title, MPN, manufacturer, datasheet URL, `standardLeadTime` (e.g. "30 周")
- `priceQuantity`: `qtyAvailable` (stock), `pricing[]` (tier prices)
- `productAttributes.attributes`: spec parameters
- `quantityTable`: alternative tier presentation

### Page-element semantics

| Field | Source | Means | Use as | Discard? |
|---|---|---|---|---|
| `qtyAvailable` | `__NEXT_DATA__` | Digikey warehouse stock | `stock_now_qty` + single 现货 row with `warehouse=None`, `ship_text=None` | — |
| `standardLeadTime` ("30 周") | `__NEXT_DATA__` | Factory lead time | Stored in `extracted.lead_time` for downstream. **Do NOT** emit as a `期货 / 工厂期货` warehouse row (page does not label it that way) | — |
| `isBackOrderAllowed` | `__NEXT_DATA__` | Whether factory orders accepted | `site_*` only | — |
| "预计美国仓库发货日期" (CMS string) | DOM | Tooltip label only — NOT a warehouse identifier | NOT a warehouse name | **DISCARD** as warehouse |
| "立即购买，当天发货" | DOM meta-description | SEO marketing copy in `<meta>` — NOT a product-page label | NOT a ship_text | **DISCARD** |
| `pricing[].unitPrice` / `breakQuantity` | `__NEXT_DATA__` | Tier prices in USD | `prices[]`. **Currency is USD** even on the `.cn` site. | — |

### Pitfalls

- **Cloudflare titles in multiple languages.** `"Just a moment"` set + 50 KB length threshold. Don't rely on either alone.
- **Three-strike fabricated labels.** Historical Digikey scraper had `DigiKey 美国仓` warehouse, `下单后立即发货` ship_text, AND a `工厂期货` row — all three were scraper inventions. **All three are now stripped.** Lead time itself is still recorded (`extracted.lead_time` carries the literal "30 周" string) but it does not appear in `stock_breakdown`.
- **`failed` status is normal.** 40 %+ of `failed` cells on a Chinese-BOM batch are genuine "Digikey doesn't carry this part" cases (search returns no `/products/detail/` link). Not a scraper bug.

## 3. HQEW (华强电子网, hqew.com)

**Script:** `scraper/scripts/scrape_hqew.py`

### Engine + why

Playwright **Chromium** `--headless=new`. The site is a **B2B aggregator** of independent supplier listings — fundamentally different model from a single-distributor site.

| Engine tried | Result | Why |
|---|---|---|
| `curl_cffi` chrome131 | ~1 KB obfuscated JS challenge (`jsjiami.com.v7`) | Custom JS obfuscator |
| Playwright Chromium | ✅ works | JS runs cleanly under a real browser |

### URL patterns

- Search: `https://s.hqew.com/<MPN>.html` (URL path, not query string)

### Detail-page reach

No detail pages — search results page IS the data. Each `<tr class="ec-data">` is one supplier listing. 12-column layout: ad-flag / supplier+badge / — / listed-MPN / brand / batch-code / quantity / package / warehouse-city / transaction-note / listing-date / —.

### Data model — different from every other source

HQEW does **not** have a unified product page. Each supplier sets their own:
- Listed MPN string (a single search returns variants — `BT168GW,115` / `BT168GW，115` (full-width) / `BT168GW 115` (space) / `BT168GW115` — all distinct rows)
- Stock quantity (their own pool)
- Brand (sometimes empty, sometimes English, sometimes CN)
- Batch code, listing date, transaction note
- **Price is private** — per-supplier prices are gated behind login + 询价 (request quote). The only public price is the top-of-page aggregate **云价格**.

### Mapping to canonical schema

| Canonical | HQEW source |
|---|---|
| `stock_now_qty` | SUM of `quantity` across the top-5 listings (cap = 5, ranked by quantity DESC across all MPN variants) |
| `stock_now_ship_text` | Constant string `"供应商现货 (具体发货请询价)"` — this is the site's own consistent phrasing for B2B listings |
| `stock_future_qty` | `null` (HQEW has no future-stock concept) |
| `stock_breakdown[]` | One row per supplier listing. `warehouse` = supplier name (e.g. `深圳市硅誉通科技有限公司（深圳）`), `quantity` = listing qty, `moq` = listing MOQ if present, extra columns: `batch_code`, `listing_date`, `remark` |
| `prices[]` | Single tier: `[{min_qty: 1, unit_price_cny: float(cloud_price), note: "云价格 (aggregate)"}]` |
| `variants[]` | Per-MPN-variant grouping. For each distinct MPN string returned by the search, one entry: `{manufacturer_part_number, listings, listing_count, top_listings_count, stock_now_qty (per-variant), ...}` |

### Variant ranking

When the search returns multiple MPN strings, sort variants by total stock DESC and pick the **canonical = first** (highest-stock variant's MPN). All variants still surface in `extracted.variants[]`.

### Page-element semantics

| Cell index | Cell content | What it means |
|---|---|---|
| 0 | Ad flag | Drop |
| 1 | Supplier name + badge | The `warehouse` field |
| 2 | (empty separator) | Drop |
| 3 | Listed MPN | The variant string |
| 4 | Brand | Manufacturer (mix of EN / CN) |
| 5 | Batch code | Production batch (yyyyMM+ format) — extra column |
| 6 | Quantity | The `quantity` field |
| 7 | Package | E.g. `SOT-223` — used for `package` field |
| 8 | Warehouse city | Appended to supplier name in parens |
| 9 | Transaction note | "原装正品" etc. — `remark` extra column |
| 10 | Listing date | "20240315+" — `listing_date` extra column |
| 11 | (empty) | Drop |

### Pitfalls

- **B2B marketplace shifts schema semantics.** Don't try to fit HQEW into a "single distributor with N stock pools" model — `warehouse` is a **supplier name**, not a physical warehouse location.
- **Variant strings vary by separator.** Treat `BT168GW,115` vs `BT168GW，115` (full-width comma) vs `BT168GW 115` (space) vs `BT168GW115` as **separate** variants — do not normalize. They may be data-entry conventions or genuine SKUs.
- **No spec parameters.** HQEW never has `parameters[]`. Empty `[]` is expected.
- **Cap matters.** Popular MPNs return 50–400 listings. Driver caps `stock_breakdown` at **top 5 by quantity** (was 30 in v1; reduced 2026-05-19 because 30 rows per chip overwhelmed cross-channel review).

## 4. Future Electronics (futureelectronics.com)

**Script:** `scraper/scripts/scrape_future.py`

### Engine + why — the HTTP/2 fingerprint wall

Future is fronted by **Akamai BMP**. Akamai's rejection target is Playwright **Chromium's** HTTP/2 frame ordering — Chromium gets `ERR_HTTP2_PROTOCOL_ERROR` on every navigation. Playwright **Firefox** has a different HTTP/2 fingerprint that Akamai allow-lists. So:

| Engine tried | Result | Why |
|---|---|---|
| `curl_cffi` various profiles | Homepage 403 (`sec-if-cpt` interstitial); other paths return SPA shells with no product data | curl_cffi can't run JS for the sensor |
| Playwright Chromium | `ERR_HTTP2_PROTOCOL_ERROR` immediately | HTTP/2 fingerprint blacklisted |
| Playwright **Firefox** | ✅ works | Firefox HTTP/2 fingerprint is allow-listed |

### URL patterns

- Search: `https://www.futureelectronics.com/search?text=<MPN>&q=<MPN>:searchRelevance`
- Detail: `https://www.futureelectronics.com/p/<category>/<mpn-slug>-<mfr-slug>-<id>` (e.g. `…/p/semiconductors--microcontrollers/stm32g030f6p6-stmicroelectronics-3158441`)

### Detail-page reach

1. Goto search URL.
2. **Dismiss cookie banner** — click any of: `button:has-text('Allow all cookies')` / `button:has-text('Accept All')` / `#onetrust-accept-btn-handler`. The banner overlays the lower-right Pricing/Stock panel; data extraction via `innerText` works through the overlay, but `_product.png` screenshots are unreadable without dismissal.
3. Collect every `a.product__list--code` on the search page — each links to a variant's detail page.
4. For each link, goto detail URL with `wait_until="domcontentloaded"`, then `page.wait_for_timeout(8_000)` for hydration, then `wait_for_function` polling for the body text to contain `"Factory Stock"` (signal that the pricing section finished rendering).
5. Re-dismiss the cookie banner after each navigation (it can re-appear).

### Data extraction — label-driven text scrape

The Future detail page has no clean SSR blob. Strategy: extract `body.innerText`, split into lines, and look up values by label.

```js
function valueAfter(label, maxAhead=4) {
    // Walk forward from the line matching `label`, skipping duplicate
    // labels and section-header lines. Return null if a section header
    // intervenes (means the original label had no value).
}
```

Labels to look up: `Global Stock`, `<Region>:` (e.g. `Singapore:`), `On Order:`, `Factory Stock:`, `Factory Lead Time:`, `Minimum Order:`, `Multiple Of:`, `Date Code:`, `Part Status:`, `HTS Code:`, `ECCN`, `Datasheet`.

### Page-element semantics

| Future label | Means | Map to canonical | Notes |
|---|---|---|---|
| `Global Stock: N` | Future's own warehouses globally | `stock_now_qty` + breakdown row `{label: "Global Stock", warehouse: "Future Electronics (global)", quantity: N}` | The chip-level 现货 |
| `<Region>: N` (e.g. `Singapore:`) | Per-region slice of Global Stock (APAC site shows Singapore) | **DROP** — per-user-rule 2026-05-19: it's a same-number duplicate of Global Stock on the APAC site; naive SUM would double-count | Singapore row suppressed unconditionally |
| `On Order: N` | Already-reserved stock | Breakdown row only when N is a non-negative integer | Suppress when value is `None` / `-1` (parse failure indicators) |
| `Factory Stock: N` | "Inventory held at our manufacturer's warehouse" (site's own definition) | Breakdown row + `stock_future_qty` = N. Future Lead Time goes into ship_text. | Suppress when N is missing/negative |
| `Factory Lead Time: N Weeks` | Mfr lead time when ordering against Factory Stock | `stock_future_ship_text` + parsed `lead_time_days` (×7) | |

### Pitfalls

- **`Factory Stock:` label appears twice on the page** — once as section heading, once as the data row. The `valueAfter` walker must skip the heading occurrence.
- **NBSP (`\xa0`) in rendered text.** Lines like `1600 per Reel` use non-breaking spaces, not regular spaces. JS `\s` matches NBSP, but a literal `' '` doesn't. Datasheet anchor on Future is ` Datasheet` (NBSP-Datasheet) — match on `/Datasheet$/`, not `' Datasheet'`.
- **`-1` is not stock.** Future occasionally renders `-1` for "unknown / not loaded" in the Factory Stock cell. `_parse_qty` returns `None` for negative values; breakdown rows only emit when `isinstance(qty, int) and qty >= 0`.
- **Currency is SGD** (Singapore Dollar) on the APAC site default. For USD, switch landing URL / set country cookie.
- **Some MPNs have zero spec parameters.** PIC16F18446T-I/SS detail page has no Technical Attributes section. `parameters: []` is a valid result, not a parser bug.
- **Multi-variant fuzzy fallback.** Same Rule-3 guard applies: ESP32-WROOM-32E-N4 search returns 7 variants including unrelated ESP32-WROVER-IE etc. Without the substring guard, picker selected wrong variants.

## 6. ONEYAC (唯样商城, oneyac.com)

**Script:** `scraper/scripts/scrape_oneyac.py`

### Engine + why

Playwright **Firefox**. Vue.js page renders client-side; raw HTML is mostly empty.

### URL patterns

- Search: `https://www.oneyac.com/search/?keyword=<MPN>`
- Detail: `https://www.oneyac.com/product/<product-id>.html`

### Detail-page reach

1. Goto search URL, wait for Vue hydration (timeout ~8 s).
2. Find the first `a[href^="/product/"]` matching the input MPN (case-insensitive, alphanumeric-substring fallback per Rule 3).
3. Goto detail URL.

### Critical: distinguish main card vs recommended-products carousel

ONEYAC's detail page has TWO price/stock tables:
- The **main product card** uses `<div class="detailPri">` for tier prices + `<span id="detail_inventory">` for stock.
- A **recommended-products carousel** below uses `<div class="c-proPri_lst">` for tier prices.

**The carousel must be ignored** — its prices belong to unrelated suggested parts. Pre-audit scraper sometimes read from `c-proPri_lst` when the main card was empty, producing wrong data (e.g. LIS2DH12TR showed stock=0 from the empty main card but tier prices from a related accelerometer).

### Page-element semantics

| Element | Means | Canonical mapping |
|---|---|---|
| `<span id="detail_inventory">N</span>` | Main product stock (0 is valid — "in catalog, OOS") | `stock_now_qty` — preserve `0` (don't conflate with `None`) |
| `<span>交期：</span><span>5天-7天</span>` (现货 form) | Marketplace fulfillment SLA | `stock_now_ship_text = "交期 5天-7天"` |
| `<span>交期：</span><span data-delivery-day-color="…">16W</span>` (OOS form) | Factory lead time when stock=0 | `stock_future_ship_text = "交期 16W"`, parse via `交期\s*(\d+)\s*W\b` → ×7 → `lead_time_days = 112` |
| `<b>起订量：</b> N` / `dynamicOrderMinNum: N` | Minimum click-to-order quantity | One input to `min_order_qty` |
| `<b>最小包：</b> N` / `minPack: N` | Minimum package quantity (the actual smallest a buyer can purchase) | The other input. Canonical `min_order_qty = max(dynamicOrderMinNum, minPack)` |
| `<div class="detailPri">` table | Tier prices for the **main product** only | `prices[]` |
| `<div class="c-proPri_lst">` table | Tier prices for **recommended carousel items** | **DISCARD** |
| 暂无价格 (literal text in `detailPri`) | No public tier prices | `site_price_state = "暂无价格"`, `prices = []` |
| Description text | Vue boilerplate ("提交 关闭 * 接收邮箱…") | Most of the captured `description_cn` is page-chrome boilerplate — best-effort only, treat with skepticism |

### Breakdown emit rule (post-audit)

| Stock state | Breakdown row |
|---|---|
| `stock_now_qty > 0` | `{label: "现货", warehouse: "唯样商城", quantity: N, ship_text: "交期 5天-7天", moq: …}` |
| `stock_now_qty == 0` AND (`site_lead_time` OR `min_order_qty` exists) | `{label: "期货", warehouse: "唯样商城", quantity: 0, ship_text: "交期 16W", moq: …}` |
| Neither stock nor lead time nor MOQ visible | No breakdown row (fallback CSV row is blank) |

### Pitfalls

- **`stock=0` is information, not absence.** ONEYAC routinely reports `stock=0` alongside `MOQ=4000` + `交期=16W`. Pre-audit code skipped the row entirely when stock=0, dropping useful MOQ + lead-time info from the CSV. Now emits as 期货 row.
- **Two-MOQ semantics.** 起订量 ≠ 最小包. When the page shows 起订量=1 but 最小包=2500, the buyer must order in 2500-unit reels. Take max.
- **Title is boilerplate.** Vue page title is generic — don't rely on it for MPN; use the H1 / detail JSON instead.
- **API value `dynamicOrderMinNum` may differ from page-visible MOQ.** Same audit principle as LCSC — visible page text is what matters.

### Status — ✅ working (session-dependent) since 2026-05-20

Originally dropped 2026-05-18 due to a domain-wide IconCaptcha gate. | Engine tried | Result | Why |
|---|---|---|
| `curl_cffi` chrome131 | ❌ 302 → `captcha.bom2buy.com` | No session cookie; captcha gate fires on every `/search`, `/parametric`, category URL |
| Firecrawl `/v2/scrape` (basic or stealth proxy) | ❌ 302 → captcha | Free-tier proxies don't solve IconCaptcha; `proxy: "stealth"` consumes 5 cr/call without bypassing |

### URL patterns

### Captcha session validation (pre-flight check)

On script startup, the script navigates to `https://www.bom2buy.com/` and checks:

```python
if "captcha.bom2buy.com" in page.url or page.title().strip() == "Captcha":
    raise CaptchaRequired(...)  # exit code 3
```

If raised, the script prints a clear user-action message ("Open Opera, navigate to bom2buy.com, solve the captcha, close Opera, re-run") and exits with **code 3** (distinct from generic failures so the batch driver can skip this source without failing the whole batch).

### Captcha session lifetime (empirical, not formally measured)

The session held across the 2026-05-20 pilot batch (10 MPNs) without re-prompting. Expected lifetime: hours to a few days based on typical captcha-cookie TTLs. If the user pre-passes the captcha just before a batch run, we expect the whole batch to complete on a single session.

### Rate limiting

- **Per-MPN delay:** 3 s between consecutive scrapes (defends against bom2buy's secondary slider-captcha that fires under sustained load).

### Mapping to canonical schema

- `manufacturer_part_number` ← variant's `data-part` attribute
- `manufacturer` ← first distributor row's mfr value (consistent across rows for the same variant)
- `stock_now_qty` ← sum of **distinct** distributors' `stock_qty` (after dedup — see below)
- `stock_breakdown[]` ← one entry per **distinct** distributor:
  `{label:'授权'|'', warehouse:<distri>, quantity:<stock>, ship_text:<region+lead>, moq:<int>, vendor_sku:<sku>, prices:[<own tier list>]}`
- `prices[]` ← cell-level top-level prices = tier list from the **cheapest authorized** distributor (CNY only). This is one summary view; the canonical per-distributor tiers live in `stock_breakdown[i].prices`.
- `datasheet_url` ← header anchor, or `null` if `javascript:void(0)`
- `lifecycle_status` ← header "生命周期"
- `min_order_qty` ← minimum MOQ across distinct distributors
- `site_distributors[]` ← full RAW per-row detail (NO dedup) — packaging variants preserved here for any downstream tool that needs the split
- `site_category` ← header "类别"

### Distributor dedup rule (CRITICAL — added 2026-05-20)

bom2buy emits multiple `tbody tr` rows for the same distributor when that distributor carries multiple **packaging variants** of the same chip (e.g. RS Components selling STM32G030F6P6 as SKUs `2396332` + `2396333` + `2396333P` — three packaging options, NOT three warehouses). For the canonical `stock_breakdown[]` we keep ONLY the first row per distributor name. Without this dedup:
- `stock_now_qty` would double-count packaging variants → inflated totals (e.g. STM32G030F6P6 went from a misleading 21-row 558 820 total to a clean 16-row 558 153 total after dedup)
- consumers reading `stock_breakdown[].warehouse` would see false "multiple warehouses per distributor"

The dropped rows are preserved verbatim in `site_distributors[]` for any tool that needs the packaging-level detail.

### Per-row prices for downstream warehouse-exploded export

Each `stock_breakdown[]` entry carries its OWN `prices` array (independent of the top-level `prices[]`). bom2buy's per-distributor tier structures differ — Digi-Key may have 9 tiers, Wuhan P&S 4 tiers, Mouser 1 single-price row at `min_qty=2000`, Avnet Americas a min_qty=160 000 factory-order. When this canonical record is fed into a warehouse-exploded `batch_index.csv` downstream (LCSC/Future/HQEW style), each per-warehouse row should derive its `min_break_qty / max_break_qty / num_price_tiers / price_at_min_qty / price_at_max_qty` from `stock_breakdown[i].prices` — NOT from the cell-level top-level `prices[]` (which would falsely repeat the cheapest authorized distributor's tiers across all rows).

## Adding a new source — checklist

When evaluating a new distributor, run this 6-step probe:

1. **Search URL probe.** `curl_cffi` chrome131 GET on the search URL. Inspect response status, body length, and look for known bot-protection signatures: Akamai `_abck`, Cloudflare `cf-ray`, jsjiami obfuscator, DataDome, PerimeterX.
2. **Homepage warmup variant.** GET homepage first, then search — many WAFs require an initial session.
3. **Engine escalation.** If curl_cffi fails: Playwright Chromium → Playwright Chromium + stealth → Playwright Firefox. Stop at the first that works.
4. **Detail page reach.** Confirm you can navigate from search results to a product detail page. **Per project rule (`feedback_test_must_reach_detail_page.md`): probe must reach the product detail page, not stop at search.**
5. **Canonical schema mapping.** Trace which page elements carry the 5 canonical fields. Run through Rule 2 (no fabricated labels) and Rule 3 (no silent wrong-variant fallback).
6. **Audit one cell manually.** Open `<MPN>_product.png` next to `<MPN>.json` and verify every JSON field has a page-visible source. Anything that doesn't gets removed.

After all 6 pass: add the source to `CHANNELS` / `CHANNEL_ORDER` / `SOURCE_LABEL` in `batch_scraper_test.py`, add a vendor-SKU mapping in `_vendor_sku()`, and add a row to `scraper_report_v3.md` § TL;DR.
