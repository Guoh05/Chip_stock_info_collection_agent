# CLAUDE.md — Chip Availability Pipeline

Cross-distributor chip availability data pipeline. For any MPN, produce a
unified view of stock, lead time, and pricing — via web scraping (`scraper/`),
first-party APIs (`api/`), or both.

## Folder map

- `scraper/` — Web-scraping per channel. Has its own `scripts/`, `doc/`, `README.md`.
- `api/` — First-party REST APIs. Same internal layout.
- `common/` — Shared code: summary renderer, merge script, **pipeline orchestrator** (`run_pipeline.py`, default entry for end-to-end runs).
- `test/` — Dev / exploratory output. Split into `scraper/`, `api/`, `merged/`, `comparison/`.
- `production/` — Production output. Mirrors `test/`: `scraper/`, `api/`, `merged/`. Drivers + the merge script accept `--env {test,prod}` (default `test`).
- `ref/` — Source-of-truth inputs (chip list xlsx, datasheets).
  - **Master chip list (2026-05-20+): `ref/Raw_chip_list_20260520.xlsx`**, sheet `Part List Modify`, column `Manufacture Part Number` (107 unique MPNs after dedup from 280 raw rows). The `Manufacture` column is reference only. (Renamed from `Shortage Emergency Response List_v2.xlsx` on 2026-05-22 for a uniform `Raw_chip_list_<YYYYMMDD>.xlsx` convention.)
  - Legacy `Chip_DataSource_Master.xlsx` (header row 4, MPN col 1, mfr col 2) still supported via the `load_chip_list` backward-compat path if explicitly passed via `--xlsx`.
- `.venv/` — Shared Python 3.10.9 venv. `.claude/` — settings, hooks, skills.

Each track's `README.md` carries the live status snapshot and run commands.
Read it before doing track-specific work.

## Hard rules (non-negotiable across tracks)

1. **Canonical schema (minimum contract).** Every per-variant record MUST
   include at least `stock_now_qty`, `stock_now_ship_text`, `stock_future_qty`,
   `stock_future_ship_text`, and `stock_breakdown` (use `null` / `0` / `[]`
   when the source has no equivalent). These five are what enables
   cross-channel comparison and must not be dropped. Records also routinely
   carry `prices`, `parameters`, `manufacturer_part_number`, `manufacturer`,
   `package`, `datasheet_url`, and source-native `site_*` keys — see each
   track's `doc/batch_output_schema.md` for the full reference. **Adding new
   fields is encouraged**; document them in the schema doc, and extend
   `common/_summary.py` if you want them rendered in the human summary.
2. **Site-native wording is preserved.** The canonical 现货/期货 scalars are
   an interpretation layer; each source's own labels live in
   `stock_breakdown[].label` and `site_*` fields and must not be renamed.
3. **MPN-variant grouping.** When a search returns multiple distinct MPN
   strings, produce one subfolder per MPN — never aggregate quantities
   across MPN strings.
4. **Output folder convention.** Every run writes to a per-run subfolder
   named `Test_<MPN>_<CHANNEL>_<ts>/` (or, inside a batch,
   `Test_<MPN>_<CHANNEL>/` under `BatchTest_<ts>/`). The output root is
   `test/<track>/` (dev) or `production/<track>/` (prod), selected via the
   driver's `--env {test,prod}` flag (default `test`). The per-run naming
   convention applies regardless of root. Never dump files loose at any
   root or share a folder across runs.
5. **READMEs auto-refresh.** The `<!-- BEGIN AUTO:status --> ... <!-- END
   AUTO:status -->` block in each track's README is regenerated by a hook
   + batch driver — do not edit it by hand. Hand-written sections of the
   README are fair game.
6. **Merge-script rules doc stays in sync.** When editing
   `common/merge_batch_for_procurement.py`, also update
   `doc/merge_for_procurement_rules.md` in the same change. A PostToolUse
   hook flags field-name drift, but prose only stays current if you re-read
   the doc each time.
7. **Pipeline orchestrator stays in sync.** When editing the CLI surface,
   output paths, preconditions, or exit-code semantics of
   `batch_api_test.py`, `batch_scraper_test.py`,
   `merge_batch_for_procurement.py`, `scrape_bom2buy.py`, or
   `_merge_bom2buy_into_batch.py`, also update `common/run_pipeline.py` and
   `doc/run_pipeline_workflow.md` in the same change. A PostToolUse hook
   flags edits to the listed files; the semantic judgment (does the
   orchestrator actually need a change?) is yours. Always run the smoke
   test from `run_pipeline_workflow.md` after pipeline-component edits.
8. **Review every new chip-list input before running the pipeline.**
   Applies to both `--env test` and `--env prod`, even under auto mode,
   even for `--limit 1` smoke tests. Don't blindly reuse a prior batch's
   cleaner script (its `MANUAL_OVERRIDES` are batch-specific literals;
   regex may miss new dirty patterns or false-positive on clean MPNs —
   treat any saved cleaner as a template, not a default). Always produce
   a cleaned chip-list xlsx that **preserves the raw `Manufacture Part
   Number` column** AND adds a new `MPN_cleaned` column; the merge
   output must then carry both raw `Manufacture Part Number` + cleaned
   `MPN_cleaned_byAgent` columns for traceability. See memory
   `feedback_input_review.md` for the full checklist (anomaly types,
   cleaning categories, what to ask the user, output column contract).

## Working scope

Default to the track the request is about. Don't modify code in another
track unless explicitly asked. `common/` belongs to both tracks — changes
there must keep the canonical schema and the summary renderer
backward-compatible.

For new work that consumes both tracks (integration, validation, final
report production), put it in `common/` or a new sibling top-level folder
rather than editing `scraper/` or `api/`.
