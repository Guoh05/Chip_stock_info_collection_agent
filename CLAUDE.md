# CLAUDE.md — Chip Availability Pipeline

Cross-distributor chip availability data pipeline. For any MPN, produce a
unified view of stock, lead time, and pricing — via web scraping (`scraper/`),
first-party APIs (`api/`), or both.

## Folder map

- `scraper/` — Web-scraping per channel. Has its own `scripts/`, `doc/`, `README.md`.
- `api/` — First-party REST APIs. Same internal layout.
- `common/` — Shared code (canonical summary renderer + cross-track utilities).
- `test/` — All output. Split into `scraper_test/`, `api_test/`, `comparison/`.
- `ref/` — Source-of-truth inputs (chip list xlsx, datasheets).
- `.venv/` — Shared Python 3.10.9 venv. `.claude/` — settings, hooks, skills.

Each track's `README.md` carries the live status snapshot and run commands.
Read it before doing track-specific work.

## Hard rules (non-negotiable across tracks)

1. **Canonical schema (minimum contract).** Every per-variant record MUST
   include at least `stock_now_qty`, `stock_now_ship_text`, `stock_future_qty`,
   `stock_future_ship_text`, and `stock_breakdown` (use `null` / `0` / `[]`
   when the source has no equivalent). These five are what enables
   cross-channel comparison and must not be dropped. Records also routinely
   carry `prices`, `parameters`, `manufacturer_part_number`, `manufacturer`,
   `package`, `datasheet_url`, and source-native `site_*` keys — see each
   track's `doc/batch_output_schema.md` for the full reference. **Adding new
   fields is encouraged**; document them in the schema doc, and extend
   `common/_summary.py` if you want them rendered in the human summary.
2. **Site-native wording is preserved.** The canonical 现货/期货 scalars are
   an interpretation layer; each source's own labels live in
   `stock_breakdown[].label` and `site_*` fields and must not be renamed.
3. **MPN-variant grouping.** When a search returns multiple distinct MPN
   strings, produce one subfolder per MPN — never aggregate quantities
   across MPN strings.
4. **Output folder convention.** Every run writes to a per-run subfolder
   named `Test_<MPN>_<CHANNEL>_<ts>/` (or, inside a batch,
   `Test_<MPN>_<CHANNEL>/` under `BatchTest_<ts>/`). The output root is
   currently `test/<track>_test/` — used for dev / exploratory / not-yet-
   trusted runs. A parallel `prod/<track>/` root (or similarly named) will
   be introduced when the pipeline graduates to production; the per-run
   naming convention applies regardless of root. Never dump files loose at
   any root or share a folder across runs.
5. **READMEs auto-refresh.** The `<!-- BEGIN AUTO:status --> ... <!-- END
   AUTO:status -->` block in each track's README is regenerated by a hook
   + batch driver — do not edit it by hand. Hand-written sections of the
   README are fair game.

## Working scope

Default to the track the request is about. Don't modify code in another
track unless explicitly asked. `common/` belongs to both tracks — changes
there must keep the canonical schema and the summary renderer
backward-compatible.

For new work that consumes both tracks (integration, validation, final
report production), create a sibling top-level folder rather than editing
`scraper/` or `api/`.
